Assigning Tasks

Lesson 3 of 10 Advanced Track: Multi-Agent Orchestration

Now that both agents are running, it’s time to give them work. The key to a successful handoff is clear boundaries and a clean handoff artifact—a shared document that one agent writes and the other reads, so both sides agree on the contract.

The Handoff Pattern

In a multi-agent workflow, you don’t want agents guessing what the other is building. Instead, you create an explicit handoff:

  1. Agent A builds its component and writes a contract document describing the interface
  2. You verify the contract looks correct
  3. Agent B reads the contract and builds its component to match

For the bookmark project, this means:

Agent Builds Handoff Role
Claude REST API in api/ Writes handoff/api-contract.md
Codex CLI client in cli/ Reads handoff/api-contract.md

The handoff/ directory is neutral territory—neither agent’s workspace, but a shared communication channel between them.

Exercise: Assign the Tasks

1 Create the handoff folder

From your control session, create the directory where the contract document will live:

mkdir -p ~/projects/bookmarks/handoff
2 Send the API task to Claude

Use send-keys to give Claude a detailed prompt. Notice the instruction to write the handoff contract:

tmux send-keys -t claude-api "Create a simple REST API for bookmarks in this directory. Use Node.js with Express. Endpoints needed: - POST /bookmarks (create: url, title, tags[]) - GET /bookmarks (list all, optional ?tag= filter) - DELETE /bookmarks/:id Store data in a bookmarks.json file. Keep it minimal. Also write ../handoff/api-contract.md with endpoint definitions, request/response JSON examples, and error formats." Enter

Claude will now start building the API and documenting the contract that Codex will need.

3 Wait for the contract, then send the CLI task to Codex

Before sending Codex its task, confirm Claude has finished the handoff document:

# Check if the contract file exists yet ls ~/projects/bookmarks/handoff/api-contract.md

Once it exists, send Codex its task with an explicit instruction to read the contract first:

tmux send-keys -t codex-cli "Create a CLI tool called 'bm' in this directory. Use Node.js. First, read ../handoff/api-contract.md and follow that contract exactly. Commands: - bm add <url> [title] [--tags tag1,tag2] - bm list [--tag tagname] - bm delete <id> It should talk to a REST API at http://localhost:3000. Keep it minimal." Enter

Timing matters: Don’t send Codex its task until the contract file exists. If Codex starts before the contract is written, it will guess at the API format instead of following it—defeating the purpose of the handoff.

4 Watch both agents work

You can monitor progress by attaching to each session temporarily:

# Attach to Claude's session tmux attach -t claude-api # (watch progress, then detach with Ctrl+J d) # Attach to Codex's session tmux attach -t codex-cli # (watch progress, then detach with Ctrl+J d)

Each time you’re done watching, press Ctrl+J d to detach back to the control session.

Pro Tip: Quick Session Switching

Instead of detaching and reattaching, use the tmux session switcher:

Ctrl+J s

This shows a list of all sessions. Use the arrow keys to highlight one and press Enter to jump directly into it. It’s much faster than the detach/attach cycle.

Method Steps Best For
Detach/Attach Ctrl+J d then tmux attach -t name Scripting, when you need to be explicit
Session Switcher Ctrl+J s then arrow keys + Enter Interactive monitoring, quick peeks
capture-pane tmux capture-pane -t name -p | tail Read-only snapshot without leaving control

Checkpoint

  • Claude is creating files in ~/projects/bookmarks/api/ and ~/projects/bookmarks/handoff/api-contract.md
  • Codex is creating files in ~/projects/bookmarks/cli/ based on the handoff contract
  • You can switch between sessions with Ctrl+J s
