Conflict Resolution

Lesson 7 of 10 Advanced Track: Multi-Agent Orchestration

When two agents work in parallel, sooner or later they’ll both reach for the same file. A shared config, a common dependency, a root-level manifest—any of these can become a collision point. In this lesson you’ll learn to establish ownership boundaries so conflicts are prevented, and how to resolve them when they do occur.

Shared Files, Competing Agents

So far, Claude and Codex have stayed in their own directories—api/ and cli/ respectively. But real projects have shared resources: configuration files, environment settings, package manifests. When you tell both agents to use a shared file, you risk one agent overwriting the other’s changes.

The solution isn’t to avoid shared files—it’s to establish clear ownership rules and enforce them through your instructions to each agent.

Exercise: Introduce a Shared Config

1 Create a shared configuration file

From your control session, create a config file at the project root that both agents will need:

cat > ~/projects/bookmarks/config.json << 'EOF' { "apiPort": 3000, "apiHost": "localhost", "version": "1.0.0" } EOF

This file lives outside both agents’ directories—it’s shared infrastructure that both sides need to read.

2 Tell both agents to use the config

Send each agent an instruction to integrate the shared config:

tmux send-keys -t claude-api "Update the API to read port and host from ../config.json" Enter tmux send-keys -t codex-cli "Update the CLI to read the API URL from ../config.json" Enter

Both agents now know about the same file. Watch what happens—will either agent try to modify the config rather than just reading it?

Exercise: Detect and Resolve Conflicts

3 Check for conflicting changes

Use git to see if both agents touched the shared file:

cd ~/projects/bookmarks git init git add -A git status

If config.json shows as modified, one or both agents changed it. Check the diff to see what happened:

git diff config.json
4 Enforce ownership with a directive

If an agent modified the config when it should have only read it, correct the behavior:

tmux send-keys -t claude-api "Read config.json but don't modify it. Only the human edits config.json." Enter

Then restore the config to your intended version:

git checkout config.json

The Rule of Ownership

For multi-agent handoff workflows, establish clear boundaries before assigning tasks. A simple ownership model:

Owner Scope Rule
Claude api/ directory Full read/write within its directory
Codex cli/ directory Full read/write within its directory
Human (you) Root config files, handoff/ Agents read only; you make all edits

State this explicitly in your prompts. AI agents don’t remember implicit rules. Every time you send a task that involves a shared file, include the boundary: “Read ../config.json for the port value but do not modify it.”

Checkpoint

  • Both agents read from the shared config file without overwriting it
  • You understand the ownership model: each agent owns its directory, you own shared files
  • You can detect conflicts with git status and resolve them by directing agents
