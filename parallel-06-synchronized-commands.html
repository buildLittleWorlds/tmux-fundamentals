Synchronized Commands

Lesson 6 of 10 Advanced Track: Multi-Agent Orchestration

So far you’ve been sending commands to one agent at a time. But when you need a status check or want both agents to act in concert, you can broadcast the same command to every session at once. This lesson covers synchronized commands, side-by-side comparison windows, and the pitfall of nested tmux sessions.

Broadcasting to Multiple Sessions

With tmux send-keys targeting individual sessions, you’ve been sending instructions one at a time. But a simple shell loop lets you send the same command to every agent session simultaneously. This is invaluable for synchronized status checks, coordinated restarts, or asking both agents to summarize their progress at the same moment.

Why not nested sessions? Your first instinct might be to open a window that attaches to another tmux session. This creates nested tmux sessions—sessions inside sessions—which leads to confusing key conflicts (your prefix key gets intercepted by the outer session). The patterns in this lesson avoid nesting entirely.

Exercise: Avoid the Nested-Session Trap

1 See why nesting is problematic

You might try creating a window that attaches to both agent sessions side by side:

tmux new-window -n agents tmux send-keys 'tmux attach -t claude-api' Enter tmux split-window -h tmux send-keys 'tmux attach -t codex-cli' Enter

This works, but now you have tmux inside tmux. Your Ctrl+J prefix is captured by the outer session, so you can’t control the inner ones without double-pressing the prefix. It gets confusing fast.

2 Use a comparison window instead

Go back to your control session’s first window, then create a clean comparison layout:

# Return to the control session's first window tmux select-window -t control:0 # Create a side-by-side comparison window tmux new-window -n compare tmux split-window -h

Now you have two panes in the control session—no nesting. You’ll use send-keys from here to talk to the agent sessions remotely.

Exercise: Synchronized Status Check

3 Send the same command to both agents

Use a for loop to broadcast a status request to every agent session at once:

# Send to both agents simultaneously for session in claude-api codex-cli; do tmux send-keys -t $session "Summarize current status in 3 bullets and list changed files." Enter done

Both agents receive the same prompt at the same time. While they respond, you can continue working in your control panes.

4 Capture and compare the outputs

After the agents have had time to respond, capture their output side by side:

echo "=== Claude ===" && tmux capture-pane -t claude-api -p | tail -10 echo "=== Codex ===" && tmux capture-pane -t codex-cli -p | tail -10

This gives you a quick snapshot of where each agent stands. You can compare progress, spot mismatches in their understanding, or confirm they’re both on track.

Timing matters: The agents won’t respond instantly. Give them a few seconds (or longer for complex prompts) before capturing output. If the response is still streaming, you’ll see a partial result—just wait and capture again.

Extending the Pattern

The for loop approach scales to any number of sessions. As you add more agents to your workflow, the same broadcast pattern works:

# Broadcast to three agents for session in claude-api codex-cli testing-agent; do tmux send-keys -t $session "Run your test suite and report results." Enter done

You can also vary the command per session while keeping the synchronized timing:

# Capture all agents into separate log files for session in claude-api codex-cli; do tmux capture-pane -t $session -p -S -1000 > ~/projects/bookmarks/${session}-snapshot.log done

Checkpoint

  • You understand why nested tmux sessions cause problems and how to avoid them
  • You can send the same command to multiple sessions using a for loop
  • You can capture and compare outputs from both agents side by side
