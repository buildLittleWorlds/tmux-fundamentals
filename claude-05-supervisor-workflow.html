The Supervisor Workflow

Lesson 5 of 10 Beginner Track: tmux + Claude Code

This lesson formalizes the two-pane pattern you started in Lesson 4 into a repeatable development rhythm. You’ll prompt Claude for features, watch it work, switch panes to test, and iterate—building out the td CLI with rapid prompt–test cycles.

The Prompt–Test Cycle

The supervisor workflow is a tight feedback loop:

  1. Tell Claude what you want (right pane)
  2. Watch it work in real time
  3. Switch panes and test the result (left pane)
  4. Report back if something’s wrong
  5. Repeat

You’re the supervisor. Claude is the implementer. The two-pane split keeps both roles visible at all times.

Exercise

1 Confirm your two-pane layout

You should have the same layout from Lesson 4: left pane for your shell, right pane with Claude Code running. If not, set it up again with Ctrl+J % and launch claude in the right pane.

2 Request the “add” command

In the Claude pane (right), type this prompt:

Add an "add" command to td.sh so I can run: ./td.sh add "Buy groceries" It should store tasks in a file called ~/.td-tasks (one per line).

Watch Claude’s output as it modifies the file.

3 Test the “add” command

When Claude finishes, move to the left pane (Ctrl+J ←) and try it out:

./td.sh add "Buy groceries" ./td.sh add "Call mom" ./td.sh add "Learn tmux" cat ~/.td-tasks

You should see the three tasks stored in the file, one per line.

4 Request the “list” command

Move back to the Claude pane (Ctrl+J →) and request another feature:

Add a "list" command that shows all tasks with line numbers.
5 Test the “list” command

Switch to the left pane and test:

./td.sh list

You should see numbered output like:

1 Buy groceries 2 Call mom 3 Learn tmux

Checkpoint

  • You can add tasks: ./td.sh add "something"
  • Tasks are stored in ~/.td-tasks
  • You can list tasks: ./td.sh list
  • Output shows numbered tasks

Why This Works

The supervisor workflow succeeds because context switching is instant. There’s no alt-tabbing between apps, no waiting for windows to load. A single key combo (Ctrl+J ← or Ctrl+J →) moves you between the two roles—directing Claude and verifying its output.

When something goes wrong: If a test fails, switch back to Claude and describe the problem. Include the error output. Claude can see the project files and fix the issue in place. This prompt–test–fix cycle is much faster than copy-pasting between separate terminal windows.

What’s Next: In Lesson 6: Windows, you’ll learn to create multiple workspaces within a single session—like browser tabs for your terminal—so you can keep Claude in one window and git operations in another.
