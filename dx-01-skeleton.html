The dx Skeleton

Lesson 1 of 10 Advanced Track: Developer Toolkit

In this lesson, you’ll create the foundation for dx—a developer experience toolkit that dispatches to AI-powered subcommands. By building the skeleton first, you establish the project structure and command-routing pattern that every future lesson will build on.

What you’ll build in this track: A developer experience toolkit called dx with commands like dx review, dx pr, dx explain, dx test, and dx standup. Each command uses tmux and AI CLIs in different ways—from non-interactive one-shot prompts to persistent background sessions. Along the way, you’ll master tmux scripting and AI automation.

The Dispatcher Pattern

The dx tool follows a common Unix pattern: a single entry-point script that routes to subcommand scripts in a commands/ directory. When you run dx review, the main script finds and executes commands/review.

This gives you a clean separation of concerns—each subcommand is its own executable file, easy to develop and test independently.

Command What It Does
dx review AI reviews your staged changes before commit
dx pr Creates a PR with AI-generated description
dx explain Explains a file or function
dx test Runs tests with AI analysis of failures
dx standup Summarizes your recent git activity

Exercise: Build the Skeleton

1 Create the project directory

Set up a dedicated directory for the dx toolkit:

mkdir -p ~/projects/dx && cd ~/projects/dx
2 Create the main dx script

This is the entry point—it parses the first argument as a command name and dispatches to the matching file in commands/:

~/projects/dx/dx
#!/bin/bash
set -e

DX_HOME="$(cd "$(dirname "$0")" && pwd)"
COMMAND="${1:-help}"
shift 2>/dev/null || true

case "$COMMAND" in
  help|-h|--help)
    echo "dx - Developer Experience Toolkit"
    echo ""
    echo "Commands:"
    echo "  dx review     Review staged changes with AI"
    echo "  dx pr         Create PR with AI-generated description"
    echo "  dx explain    Explain a file or code section"
    echo "  dx test       Run tests with AI failure analysis"
    echo "  dx standup    Summarize recent git activity"
    echo ""
    echo "Options:"
    echo "  --ai=claude   Use Claude (default)"
    echo "  --ai=codex   Use Codex"
    ;;
  *)
    SUBCMD="$DX_HOME/commands/$COMMAND"
    if [[ -x "$SUBCMD" ]]; then
      exec "$SUBCMD" "$@"
    else
      echo "Unknown command: $COMMAND"
      echo "Run 'dx help' for usage"
      exit 1
    fi
    ;;
esac

Make the script executable:

chmod +x dx

Key details: DX_HOME resolves to the script’s own directory (so subcommands can find each other regardless of where you call dx from). The shift removes the command name, passing remaining arguments to the subcommand.

3 Create the commands directory

This is where all subcommand scripts will live:

mkdir -p commands

Your project structure now looks like this:

~/projects/dx/ ├── dx # Main dispatcher script └── commands/ # Subcommand scripts go here
4 Test the skeleton

Verify both the help output and the error handling for unknown commands:

./dx help

You should see the full list of planned commands. Now try an unknown command:

./dx unknown

This should output Unknown command: unknown followed by the help hint. Both paths working means your dispatcher is ready.

Checkpoint

  • ./dx help shows the list of available commands
  • Unknown commands display an error message with a help hint
  • The commands/ directory exists and is ready for subcommand scripts
