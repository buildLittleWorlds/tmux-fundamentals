The Review Command

Lesson 3 of 10 Advanced Track: Developer Toolkit

Time to build your first real subcommand. dx review sends your staged git changes to an AI for feedback before you commit—catching bugs, security issues, and style problems in a single step. This is a non-interactive command: it runs the AI, prints the response, and exits.

Non-Interactive AI Commands

dx review is the simplest pattern in the toolkit: gather context from the local environment (in this case, git diff --cached), build a prompt, pass it to ai_prompt, and print the result. No tmux sessions, no interactivity—just a one-shot question and answer.

This pattern works well for commands where the AI’s job is to analyze something and report back. The user reads the output and decides what to do next.

Subcommand anatomy: Every dx subcommand follows the same opening pattern—resolve the project root, source lib.sh, and call parse_ai_flag. After that, the command is free to do its own thing.

Exercise: Build the Review Command

1 Create the review command

This script goes in the commands/ directory. It grabs the staged diff, constructs a review prompt, and sends it to the AI:

~/projects/dx/commands/review
#!/bin/bash
set -e

SCRIPT_DIR="$(cd "$(dirname "$0")/.." && pwd)"
source "$SCRIPT_DIR/lib.sh"
parse_ai_flag "$@"

# Get staged diff
DIFF=$(git diff --cached)

if [[ -z "$DIFF" ]]; then
  echo "No staged changes to review."
  echo "Stage changes with: git add <files>"
  exit 1
fi

echo "Reviewing staged changes with $AI_CLI..."
echo ""

# Create the prompt
PROMPT="Review this git diff for a code review. Be concise. Point out:
1. Any bugs or issues
2. Security concerns
3. Style/readability improvements
4. Missing error handling

If everything looks good, just say 'LGTM' with a brief note.

\`\`\`diff
$DIFF
\`\`\`"

# Run the AI
ai_prompt "$PROMPT"

Make it executable:

chmod +x commands/review

Key details: SCRIPT_DIR navigates up one level from commands/ to find lib.sh. The guard clause exits early with a helpful message if nothing is staged. The prompt asks for four specific categories of feedback, keeping the AI focused.

2 Test it with staged changes

Initialize a git repo in the dx project (if you haven’t already) and stage the files you’ve built so far:

cd ~/projects/dx git init git add dx lib.sh

Now run the review command:

./dx review

The AI will analyze your staged diff and print feedback. Since these are the skeleton and library files, you should see something like “LGTM” with minor notes.

3 Try with a different AI

Verify that the --ai flag from lib.sh works end-to-end:

./dx review --ai=codex

You should see Reviewing staged changes with codex... followed by the Codex CLI’s response. Same diff, different AI—the abstraction layer is working.

Your project structure now looks like this:

~/projects/dx/ ├── dx # Main dispatcher script ├── lib.sh # Shared AI functions └── commands/ └── review # AI-powered code review

Checkpoint

  • dx review shows AI feedback on staged changes
  • Works with both Claude and Codex via the --ai flag
  • Exits cleanly after showing the review (non-interactive)
